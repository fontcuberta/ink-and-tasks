# Database Audit — Ink & Tasks
**Date:** 2026-03-05

## Summary
Core schema is well-designed with proper uuid PKs, CHECK constraints on enums, RLS enabled on every table. However, `todos.user_id` is nullable without cascade, `project_images` is missing an UPDATE RLS policy (breaking "set primary"), `reference-images` bucket is publicly readable, and no indexes exist.

## Findings
**Critical — 6 | Suggestion — 6 | Nice to have — 4**

### Critical
1. **`todos.user_id` is nullable** — orphan rows possible, pre-auth data loss risk
2. **`todos.user_id` FK missing `ON DELETE CASCADE`** — inconsistent with other tables
3. **`project_images` missing UPDATE RLS policy** — `setPrimary()` silently fails
4. **`reference-images` bucket is publicly readable** — client reference photos exposed
5. **No file size limit on storage uploads** — unbounded storage costs
6. **No MIME type restriction on uploads** — dangerous file types uploadable

### Suggestions
7. No indexes on any table — full table scans on every RLS check
8. `created_at` missing `NOT NULL` on newer tables
9. `consent_forms` missing UPDATE RLS policy
10. `ProjectModal.vue` — missing error handling on deleteImage/setPrimary
11. `QuickAdd.vue` — image record insert missing error handling
12. `setPrimary` uses two non-atomic updates

### Nice to have
13. Initial `todos` migration missing `IF NOT EXISTS`
14. Projects migration creates two tables in one file
15. Duplicate auth module (`src/auth.js` + `useAuth.js`)
16. `consent_forms` has no separate `created_at` column

## What's solid
- UUID PKs with `gen_random_uuid()` everywhere
- RLS enabled on every table with correct `WITH CHECK` on INSERT
- CHECK constraints on all enum-like fields
- Storage upload paths use `user.id` prefix
- Explicit `.order()` on every list query
- User-visible toast feedback on all major operations
