# UX Audit — Ink & Tasks
**Date:** March 3, 2026
**Environment:** https://inkandtasks.netlify.app

## Summary
The app is solid and working end-to-end, but has 6 failures and 2 warnings. The design system, interaction flows, and overall aesthetic are well structured — the issues are mostly accessibility gaps and silent error handling.

---

## Findings

### 🔴 Critical — 4 issues

#### 1. All 4 database operations fail silently
**Domain:** Interaction
**Observed:** `loadTasks`, `addTask`, `completeTask`, `deleteTask` all call `console.error` only — the user sees nothing if a network or permissions error occurs.
**Fix:** Replace silent `console.error` with `showToast('Something went wrong', { type: 'error' })`.

---

#### 2. Double-click on complete/delete fires two database calls
**Domain:** Interaction
**Observed:** The task list click handler has no guard. A double-click before the first `await` resolves sends two concurrent Supabase calls for the same task.
**Fix:** Use an `inFlightIds` Set to guard concurrent operations; disable pointer events on the item while in flight.

---

#### 3. Main task input has no `<label>`
**Domain:** Accessibility
**Observed:** `#task-input` has no `<label for="task-input">` — it relies on placeholder text only, which screen readers do not treat as a label.
**Fix:** Add `<label class="sr-only" for="task-input">Task</label>` (visually hidden).

---

#### 4. No focus trap inside the auth modal
**Domain:** Accessibility
**Observed:** `openModal()` focuses `#auth-email` on open, but Tab will eventually move focus outside the modal into the page behind the backdrop.
**Fix:** Add a `trapFocus` keydown handler on the overlay that cycles Tab/Shift+Tab within the modal's focusable elements only.

---

### 🟡 Suggestion — 4 issues

#### 5. Auth error message not announced to screen readers
**Domain:** Accessibility
**Observed:** `<p id="auth-error">` has neither `role="alert"` nor `aria-live`. It appears visually but screen readers will not announce it.
**Fix:** Add `role="alert"` to the element.

---

#### 6. Empty-state SVG not hidden from accessibility tree
**Domain:** Accessibility
**Observed:** The inline `<svg class="empty-illustration">` has no `aria-hidden="true"`. Screen readers will attempt to read the SVG path data as content.
**Fix:** Add `aria-hidden="true"` to the SVG element.

---

#### 7. 10+ hardcoded hex values outside `:root`
**Domain:** Visual consistency
**Observed:** `#d97070` appears 5 times alone. `#d9b85c`, `#6aab84`, `#141008` and others are scattered throughout. This breaks the design token contract.
**Fix:** Add `--color-danger-text`, `--color-success-text`, `--color-gold-hover`, `--color-header-dark`, and badge-specific tokens to `:root`.

---

#### 8. Submit button not re-enabled if `addTask()` throws unexpectedly
**Domain:** Interaction
**Observed:** No `try/finally` around the submit handler's async block. An unhandled exception permanently disables the Add button until page reload.
**Fix:** Wrap async block in `try/finally { btn.disabled = false }`.

---

### 🟢 Nice to have — 1 issue

#### 9. `task.priority` is unescaped in innerHTML
**Domain:** Security / Interaction
**Observed:** Priority is read from the database and interpolated directly into `innerHTML` without `escapeHtml()`. The form `<select>` constrains it at input, but a direct DB write could inject HTML.
**Fix:** Call `escapeHtml(task.priority)` before both interpolations.

---

## What's working well
- All icon-only buttons have correct `aria-label` attributes
- Modal has `role="dialog"` + `aria-modal="true"`
- `escapeHtml()` correctly applied to task text, notes, and user email
- Escape key and click-outside both close the modal
- Toast border colours use CSS custom properties (no hardcoding)
- Auth flow tab switching correctly clears form and errors
- Overdue detection logic is correct and scoped to incomplete tasks only
- All form inputs (except main task input) have matching `<label>` elements
